# Code Style Rules

## Formatting

- **4-space indentation**, no tabs (`"indent": ["error", 4]`, `"no-tabs": "error"` in ESLint)
- ESLint extends `@typescript-eslint/recommended` + `react/recommended` + `react-hooks/recommended`
- `react/react-in-jsx-scope` is off — no need to import React in JSX files
- `@typescript-eslint/no-explicit-any` is off — `any` is allowed but avoid it in public API types

## TypeScript

- Target: ES2020, module resolution: `bundler`
- Strict mode is on — no implicit `any`, no implicit returns, strict null checks
- The library ships its own `.d.ts` types (generated by `vite-plugin-dts`); do not use `@types/` workarounds for peer deps

## Naming

- React components: PascalCase
- Hooks: `use` prefix, camelCase
- Test ID constants in `dataTests.ts`: nested object, camelCase keys (e.g. `dataTests.filters.textInput`)
- Exported types from `table.extension.ts` follow TanStack Table naming conventions

## File organization

- One component per file; file name matches the exported component name in camelCase (e.g. `tableHeader.tsx` exports `TableHeader`)
- Barrel `index.ts` files re-export from subdirectories; update the relevant barrel when adding a new component
